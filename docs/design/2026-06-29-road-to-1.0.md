# HIRIS — Road to 1.0 (rimozione flag `experimental`)

**Stato di partenza:** v0.19.0, `stage: experimental` in `hiris/config.yaml`.
**Tesi:** codice e feature sono già di livello 1.0 (731 test, CI, audit semaforo
superato, UI mobile a posto). Il gap per togliere `experimental` è ~80%
**operativo/validazione + sicurezza-dati + documentazione**, non sviluppo di feature.

Il 1.0 si dichiara quando TUTTI i blocker sotto sono ✅. Ogni item ha criterio di
accettazione verificabile.

---

## Blocker 1 — Soak / rodaggio reale  ⏳ (tempo)
Le capability costruite a giugno 2026 (storico, cattura `state_changed`, digest
notturno, `get_history` via MCP, apply reale proposte) sono verdi sui test ma non
ancora rodate sul campo.

- [ ] 0.19.0 in esecuzione reale su HA per **≥ 3–4 settimane** senza crash né
      perdita dati.
- [ ] Verificare almeno una volta in reale, end-to-end:
  - [ ] cattura: abilita una categoria in *Storicizzazione* → dopo qualche giorno
        `get_history` restituisce bucket reali.
  - [ ] digest: dopo ~1 settimana compaiono insight settimanali (recall_knowledge).
  - [ ] compact notturno gira senza errori (log) e il DB resta limitato.
  - [ ] proposta automazione: Claude propone → *Attiva* → l'automazione **esiste e
        gira** in HA.
  - [ ] semaforo: azione gialla → notifica iPhone → approva → esegue; rossa →
        conferma in HIRIS.
- [ ] Raccogliere i problemi emersi e chiuderli.

## Blocker 2 — Sicurezza dei dati negli upgrade  🔴 (codice)
Gli store SQLite (`knowledge`, `history`, `proposals`, `memory`, `vault`, `chat`)
usano solo `CREATE TABLE IF NOT EXISTS` — **nessuna migrazione** (verificato: zero
`ALTER TABLE`/`PRAGMA user_version`). L'addon si auto-aggiorna: il primo cambio di
schema rischia i dati esistenti.

- [ ] Introdurre versioning schema (`PRAGMA user_version`) + funzione di
      migrazione idempotente per ogni store (no-op se già alla versione corrente).
- [ ] Test: apertura di un DB "vecchio" (schema N-1) → migra senza perdere dati.
- [ ] Documentare backup/restore: gli snapshot HA coprono `/data`; indicare cosa
      contiene (incl. `vault.db` con segreti) e come ripristinare.

## Blocker 3 — Postura di sicurezza firmata  🟡 (security + doc)
L'audit adversariale del semaforo (2026-06-29) ha retto; gli item principali sono
chiusi in 0.18/0.19. Residui:

- [ ] Granularità **per-entità** sui domini eterogenei (`switch`, `script`,
      `scene`, `cover`) — oggi il verde è per-dominio (tutto o niente). Implementare
      o **accettare-e-documentare** esplicitamente il limite.
- [ ] `pip-audit` / scan CVE: eseguire e confermare pulito (o triage documentato).
- [ ] Documento "Modello di sicurezza" per l'utente: cosa fa il semaforo, cosa
      espone il gateway al cloud, raccomandazione "tieni lock/alarm/cover/siren NON
      verdi", isolamento di Claude dalla policy/kill-switch.
- [ ] Decisione esplicita sul default di `CONFIRM_ACTIONS` per chi installa da zero.

## Blocker 4 — Produzionizzazione del gateway MCP  🟡 (ops)
È il componente che espone la casa al cloud, ma è il meno "stabile" operativamente:
repo **local-only senza remote GitHub**, deploy manuale per copia file sul `.31`,
segreti solo lì, niente CI.

- [ ] Repo gateway tracciato su GitHub (privato) + CI (pytest).
- [ ] Backup sicuro dei segreti (`.env`, `oauth_key.pem`) fuori dal solo `.31`.
- [ ] Runbook deploy/rollback + healthcheck (`/mcp`→401, oauth-meta→200).
- [ ] Allineare versione gateway alla release HIRIS (oggi non versionato).

## Blocker 5 — Documentazione allineata  🟡 (doc)
I doc (`docs/`, bilingue) non coprono le novità.

- [ ] Aggiornare/aggiungere: storico & `get_history`, pagina *Storicizzazione*,
      MCP gateway + semaforo, flusso proposte (proponi→attiva), second brain/digest.
- [ ] Guida "primo avvio sicuro" (token, semaforo, cosa NON mettere verde).
- [ ] CHANGELOG: sezione riassuntiva 1.0.

---

## Cosa NON serve per 1.0 (evitare over-engineering)
Nuove feature; più test unitari fini a sé stessi; riscritture. Feature complete,
suite solida, CI presente, mobile a posto, semaforo ben progettato e hardenato.

## Sequenza consigliata
1. (fatto) 0.19.0 live + redeploy gateway.
2. **Blocker 2** (migrazioni DB) — codeabile subito, sblocca la sicurezza upgrade.
3. **Blocker 3 + 5** (security doc + pip-audit + doc) in parallelo al soak.
4. **Blocker 4** (gateway in CI/remote).
5. **Blocker 1** (soak ≥3–4 settimane) corre in parallelo a tutto.
6. Quando tutti ✅ → rimuovere `stage: experimental` → **1.0.0**.
